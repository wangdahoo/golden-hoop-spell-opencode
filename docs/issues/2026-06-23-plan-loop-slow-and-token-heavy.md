# 诊断：为什么一个简单需求跑了 ~75 分钟、5 轮、483KB

> 触发案例：`cases/session-ses_10c4.md`
> 最终产出方案：`.ghs/plans/2026-06-23-plan-truncation-save-fix-v5.md`
> 诊断日期：2026-06-23
> 性质：问题梳理 + 根因分析（不含代码改动）

## 一、事实还原（这个会话到底发生了什么）

| 指标 | 值 |
|---|---|
| 会话耗时 | ~75 分钟（17:01 → 18:16） |
| 会话体积 | 6125 行 / 483KB |
| 评审轮数 | **5 轮**（`max_rounds=5`，0 breaches，踩着上限险过） |
| 最终方案 | v5，108 行 / 6.6KB，3 个 Phase、动 4 个文件——**极简单** |
| 各轮裁决 | R1 FAIL(3S+6M) → R2 FAIL(2S+2M) → R3 FAIL(1S+0M) → R4 FAIL(1S+2M) → R5 PASS |

**关键事实：R1/R2/R3 三轮完全不是因为设计分歧而 FAIL，而是因为方案正文本身在传输中被截断**：

- R1：落盘方案仅 **979 字节 / 13 行**（半句话、反引号未闭合），`status` 却已是 `reviewing`
- R2：**4537 字节 / 48 行**，只剩修订日志，设计正文全丢
- R3：**2492 字节 / 46 行**，只写完 Phase 1

reviewer 三次都给出同一类 Severe：*"待评审正文本身被截断，无法作为完整方案评审"*。**直到 R4 方案正文才第一次完整落盘**——而那时主 AI 已经放弃了 designer 的设计职责（见第三节）。

## 二、根因（按影响排序）

### 根因 #1（决定性）：输出传输通道被截断，破坏了 subagent → 主 AI 的数据通路

这是吞噬 60% 轮次的元凶，且极具讽刺意味——**这个方案本身就是用来修截断 bug 的，而截断 bug 阻止了方案被写出来**。

故障链（已由 `.ghs/plans/2026-06-23-plan-truncation-save-fix-context.md:171` 自己确认）：

1. designer subagent 产出了**完整**方案（包在 `<<<PLAN_START>>>...<<<PLAN_END>>>` 里）。
2. Task 工具把结果返回主 AI 时，被 **OpenCode 运行时显示层截断**（会话实证：`...14045 bytes truncated...` / `...4055 bytes truncated...`）。完整原文其实存到了 tool-output 文件，但主 AI 只看到截断版。
3. 主 AI 把**截断文本**原样作为 `plan` 参数传给 `ghs-plan-review`。
4. parser 的后果分两种：
   - **plan 模式**：`whole_body` 策略对截断正文仍返回 `fallback_used`（>200 字符），handler **静默 persist 残缺正文 + advance**（`plan-review.ts` 只在 `empty/malformed` 才重试）→ reviewer 拿到半份方案 → FAIL。
   - **review 模式**：截断砍掉尾部 `REVIEW COMPLETE | Verdict:...` → `verdict===null` → 重试。
5. 反馈死循环：截断 → 残稿落盘 → reviewer FAIL → revise → designer 再生成（再被截断）→ 重复。

### 根因 #2：三角色 dispatcher「全量重生成」放大每轮开销

每轮 = designer **从头重写整个方案** + reviewer **从头重写整个评审**，没有 diff/增量。任何一处小改（R4 其实只改了"删一个反引号信号 + 改个工具名"）都要双方全部重跑，且**每次都要再过一遍那个有损通道**。简单方案本该 1-2 轮收敛，架构却把它摊到 5 轮的全量重生成上。

### 根因 #3：上下文膨胀——读了个 728KB 的会话文件

需求是"分析 `session-ses_10d0.md`"，该文件 **728KB**。context-explorer 把这个巨型文件读进上下文，会话前 ~2000 行大量是它的引用/转述，**拖大了后续每个 subagent 的 prompt 成本**。这是"慢且费 token"的直接来源之一。

### 根因 #4：缺少截断韧性的传输设计

整个设计依赖**大段结构化文本一次性穿过有损通道**、靠分隔标记定界。没有：

- "subagent 把输出写到文件、只回传路径"的模式（可完全绕开显示层截断）；
- finalize 完整性护栏（**这正是该方案最终提议要加的——但发现它的代价是 5 轮**）。

唯一真正有效的恢复手段（主 AI 读 tool-output 文件尾部补全标记）是**临时的、未内置进流程**的（见 `context.md:171`、`ses_10d0` 的 line 3457/4950）。

## 三、工作流退化信号（最值得警惕）

到 R4/R5，主 AI 彻底绕开了 designer 的设计职责（`cases/session-ses_10c4.md:5174-5206`）：

> 主 AI 自己写好整份方案，给 designer 的 prompt 变成：**"你的唯一任务：逐字输出以下文本，不做任何修改。不要读任何文件。不要做任何分析。直接输出以下文本…"**

designer 沦为**规避截断通道的哑中继**，R4/R5 的"设计"是主 AI 自己做的。这说明：**当前三角色架构对"需要较长输出"的任务基本不可用**——它不是在帮主 AI，而是在跟传输通道搏斗。

## 四、量化小结

- **时间**：~75 分钟。
- **轮次**：5 轮，其中 R1/R2/R3（60%）为纯损耗（零设计分歧，纯截断）。
- **Token**：会话 483KB；上游还消化了 728KB 的 `ses_10d0`，prompt 被显著放大。
- **产出**：v5 方案 6.6KB / 108 行 / 3 Phase / 4 文件——本质简单。

## 五、结论

简单需求之所以又慢又费 token，**不是设计本身有多复杂，而是 ghs 把方案/评审这种大文本塞进了一条会截断的 subagent 返回通道，且没有文件化传输 + 完整性护栏兜底**。3/5 轮是纯损耗（零设计分歧），上下文膨胀和全量重生成又把每轮成本乘了上去。

## 六、潜在修复方向（仅记录，未实施）

1. **文件化传输**：subagent 把结构化输出写到磁盘，只回传文件路径，主 AI 再读盘——彻底绕开显示层截断（最高杠杆）。
2. **finalize 完整性护栏**：拒绝把疑似截断（未闭合围栏 / 长度过短）的 `plan_content` 写入磁盘与 git（保证性兜底）。
3. **malformed/截断恢复指引**：`buildRetryInstruction` 命中截断签名时，引导主 AI 读 tool-output 文件尾部取回完整原文后重传。
4. **增量评审**：designer/reviewer 改为基于上一版的 diff/局部修订，而非全量重生成，降低每轮 token 与重过有损通道的概率。
5. **上下文瘦身**：对超大输入文件（如 728KB 会话记录）先做摘要/切片，避免整体灌入 subagent prompt。

> 注：方向 1-3 恰好就是 v5 方案本身的内容——也就是说，**发现并解决这个问题花了 75 分钟，而答案正是"如何避免再花这 75 分钟"**。
